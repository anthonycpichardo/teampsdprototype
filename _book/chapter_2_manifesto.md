<p><strong>Team PSD Scientific Values</strong> guide additional <strong>Participatory</strong> and <strong>Open Science</strong> principles:</p>
<ul>
<li><strong>Participatory Research</strong> encourages us to <strong>co-create our scientific research</strong>. Therefore...</li>
<li>We <strong>share decisions</strong>, which requires a high level of <strong>documentation.</strong></li>

<li>We strive use <strong>transparent</strong> and <strong>accessible</strong> processes and platforms, and develop <strong>transparent, accessible resources</strong>.</li>
</ul>
<h2><a id="user-content-guiding-principles" class="anchor" aria-hidden="true" href="#guiding-principles"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Guiding Principles</h2>
<h3><a id="user-content-open-source-transparent-reproducible" class="anchor" aria-hidden="true" href="#open-source-transparent-reproducible"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Open-Source, Transparent, Reproducible</h3>
<p>We value an open-source, transparent &amp; reproducible workflow.</p>
<ul>
<li>Most of our work is public-facing with the exception of any items that include Protected Health Information (PHI) or Personally Identifiable Information (PII). All of our public materials are free to download and use. We want our insights and resources to benefit the larger community.</li>
<li>We use R, a free open-source coding language and a specific file naming convention to ensure all of our matierals are machine &amp; human readable (all lower case, no spaces, with dates as yyyy_mm_dd i.e. teampsd_guiding_principles_2020_01_01).</li>
<li>Use email for any private discussions. Host any private files in password-protected locations or in folders behind the VA firewall. When in doubt, ask an HQ member or err on the side of caution.</li>
<li>Make sure your work and accompanying documentation allows other team members or scientists in the field to reproduce and understand your work without further questions.</li>
</ul>
<h3><a id="user-content-high-visibility" class="anchor" aria-hidden="true" href="#high-visibility"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>High Visibility</h3>
<p>Our work has high visibility.</p>
<ul>
<li>Keep in mind we work under the federal government of the United States for the Veterans Health Administration (VHA), the largest integrated healthcare system in the country. Everything we produce is a reflection of the VHA.</li>
<li>We work with a wide range of nationally-distributed partners both internal and external to VA, including very important and high-level stakeholders. Double check the role and responsibility of people you are communicating with before you do.</li>
<li>With these partners, we work in a participatory learning manner and iterate based on feedback from the field to ensure our work is responsive to ongoing changes.</li>
</ul>
<h3><a id="user-content-team-time" class="anchor" aria-hidden="true" href="#team-time"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Team Time</h3>
<p>Any time saved is team time.</p>
<ul>
<li>Ask questions early and often to prevent escalating issues down the road. Refer to existing resources (cheatsheets, checklists, etc.) as well for clarification.</li>
<li>Double check all work before handing it off to the next team member to reduce rework.</li>
<li>Think through dependencies across the team and partners and prioritize work based on the most recent information you have.</li>
<li>Manage workflow asynchronously (via GitHub) and only schedule meetings when absolutely necessary.</li>
</ul>
<h3><a id="user-content-communication" class="anchor" aria-hidden="true" href="#communication"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Communication</h3>
<p>Use effective communication (across all types of communication including emails, GitHub, Lucid, etc.).</p>
<ul>
<li>Assume everyone you’re communicating with is smarter than you and cares more than you and is busier than you.</li>
<li>Use clear and concise language, with formatting (bold, underline, bullets, etc) to emphasize the main decisions or issues making sure to include the "Who" "What" and "When".</li>
<li>Always include the full context and details necessary to make an informed decision (and make sure you are up to speed on the context and details before responding).</li>
<li>Use complete sentences as much as possible and write in the active voice for clarity</li>
<li>Often the team looks back to previous records of meetings, GitHub discussions, etc. to remember and track decisions that were already made or if they missed the meetings where an issue may have been discussed. As such, we always need to keep the most accurate record possible to reduce rework and provide clarity.</li>
<li>Use emojis or humor (as appropriate) to help maintain a positive and collegial vibe. Turn on your webcam as much as possible for the face-to-face interaction.</li>
</ul>
<h3><a id="user-content-active-listening" class="anchor" aria-hidden="true" href="#active-listening"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Active Listening</h3>
<p>We use active listening skills to ensure understanding and accurate tracking.</p>
<ul>
<li>We work daily with team members and partners that are experts in their respective fields, and it is easy to lose track of a complex discussion.</li>
<li>We've found that reflecting-back requests and decisions in your own words has been the best method to reduce miscommunication and to keep track of all of our decisions or issues accurately. If you ever need help while scribing, always "tap" someone else on the team for assistance.</li>
<li>Ask for clarification and slow down if necessary.</li>
</ul>
<h4><a id="user-content-active-listening-is-a-stance" class="anchor" aria-hidden="true" href="#active-listening-is-a-stance"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Active listening is a stance.</h4>
<p>Taking the stance that misunderstanding is the norm and using skills appropriate to that reality.</p>
<ol>
<li>General skills: Reflecting content for efficiency and interpersonal rapport (i.e., avoiding rework and frustration)</li>
</ol>
<ul>
<li>"When in doubt, check it out" - It's the listener's job to help the speaker understand what they are getting or what they missed.</li>
<li>Let the speaker know when you're falling behind. Stop them before the conversation exceeds your understanding.</li>
<li>Use the same precise language/vocabulary, esp. Team PSD or <em>MTL</em> terms with specific meanings.</li>
</ul>
<ol start="2">
<li>"Chunking" components of what someone said (e.g., listen for the "and" when someone is speaking)</li>
</ol>
<ul>
<li>Listen for the main points.</li>
<li>This is a key skill to keep up with complex ideas in the moment (You don't want to get bogged down on point 1 and miss point 2).</li>
</ul>
<ol start="3">
<li>Reflection of feeling - listen for the feeling words and reflect them exactly.</li>
</ol>
<ul>
<li>Using the exact feeling word you heard is the safest way to ensure the person feels heard.</li>
<li>Don't reflect "frustration" when someone is expressing "disappointment" or "stress."</li>
<li>You can use these for non-verbals too.</li>
<li>If you hear a feeling, it's best to address it in your reflection.</li>
</ul>
<ol start="4">
<li>Synthesize two person's ideas - or just two ideas (or more)</li>
</ol>
<ul>
<li>e.g. "on the one hand, I hear Anthony saying..." and "on the other hand, I hear Stacey saying..."</li>
<li>Synthesis brings the ideas together. It is not just active listening both ideas as separate ideas.</li>
</ul>
<ol start="5">
<li>Summarize - used when it's time to wrap up, and move on, and a lot of ideas have been said.</li>
</ol>
<ul>
<li>This is not the time to bring up new points. If you do have something to add, say it first.</li>
<li>It's best to end with a summary of the key points of consensus and the key take aways of what to do next.</li>
</ul>
<ol start="6">
<li>After you reflect, add to what the person is saying (a key to improv is "yes, and..." - don't reject ideas, just try to understand first and then build on the ideas).</li>
</ol>
<ul>
<li>Forward momentum comes from making sure you understand the person and add your own contribution.</li>
<li>Listening is about being present to understand; you do not have to agree with what a person is saying.</li>
<li>Before you even get to agreement/disagreement, make sure you have a shared understanding first.</li>
<li>KEY IDEA: Really attend to make sure you understand. Do NOT think about your response when you should be listening.</li>
</ul>
<h4><a id="user-content-things-to-look-out-for-when-listening" class="anchor" aria-hidden="true" href="#things-to-look-out-for-when-listening"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Things to look out for when listening:</h4>
<p>Know thyself:
You are the only one who knows whether it's time to multi-task, focus, scribe/document as a form of listening, or whether typing would be a distraction.</p>
<p>Repetition cycle:
If the speaker you're listening to keeps repeating something, your first move should be to assume that you're missing something and ask what it is.</p>
<p>Your own feelings:
If you're starting to feel frustrated, it's a key sign that you should use active listening to get back on the same page.</p>
<h2><a id="user-content-project-management-principles-flowmap-needed" class="anchor" aria-hidden="true" href="#project-management-principles-flowmap-needed"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Project Management Principles [flowmap needed]</h2>
<ul>
<li><strong>NOTE: Team PSD Scientific Values guide <em>how</em> and <em>why</em> we synthesize the approaches below</strong></li>
</ul>
<p><strong>Team PSD integrates Waterfall</strong> principles because:</p>
<ul>
<li>We are a <strong>research team</strong> that cannot deviate in quality, scope, timeline or cost from our thoroughly developed and vetted <strong>research plan</strong></li>
<li>Therefore, there are several <strong>well-defined scientific requirements</strong> where change cannot occur - unless the PI and co-I senior scientists advise an operational definition or implementation detail is within the scope of the federally funded and registered, scientifically peer-reviewed, and VA ORD and Stanford IRB reviewed procedures.</li>
</ul>
<p><strong>Team PSD integrates Agile</strong> principles because:</p>
<ul>
<li>Our research team’s method is a an implementation of <strong>participatory system dynamics</strong> in a national production environment.</li>
<li>These methods <strong>historically have iterated in-person</strong> locally.</li>
<li>Our innovation is local priorities, local data and local insights at national scale, made possible using <strong>online platforms</strong> and <strong>virtual processes</strong>.</li>
</ul>
<p><strong>Team PSD integrates Waterfall and Agile</strong> principles because:</p>
<ul>
<li>We are using a funded <strong>a priori research design</strong> (Waterfall) to <strong>study</strong> an <strong>implementation science</strong> (Agile) problem.</li>
<li>Producing generalizable knowledge requires implementation scientists to <strong>follow and report publicly</strong> progress completing a <strong>rigorous, replicable plan</strong> (Waterfall)</li>
<li>However, the context is not a tightly controlled laboratory, rather we are <strong>working with operational partners</strong> to <strong>implement our innovation</strong> the dynamic context of real world healthcare operations (Agile)</li>
</ul>
<p><strong>Team PSD integrates Lean</strong> principles because:</p>
<ul>
<li>Team PSD <strong><em>processes</em></strong> are assessed for continuous improvement (kaizen/muda elimination)</li>
<li>Team PSD <strong><em>communication</em></strong> is designed to reduce rework (muda/waste)</li>
<li>Team PSD encourages <strong><em>minimum viable products</em></strong> to find and synthesize team and partner expertise (just-in-time)</li>
<li>Team PSD <strong><em>platforms</em></strong> are designed to increase visual production controls (kanban/signboard; andon alerts)</li>
<li>Note: VA uses Lean as a primary approach to healthcare quality improvement.</li>
</ul>
<p><strong>Team PSD integrates Scrum</strong> principles because:</p>
<ul>
<li>We need to be able keep work produced by our t<strong>ransdisciplinary research team</strong> on the same page with our <strong>cross-functional team</strong>.</li>
<li>Therefore, we use <strong>sprints</strong> (epics/milestones)so that the <strong>team priorities</strong> can be aligned/re-aligned efficiently.</li>
<li>We also use <strong>workgroup leads</strong> (scrum masters), <strong>workgroups</strong> (owners) and <strong>workgroup</strong> meetings to benefit from the efficiency of <strong>divvying up/delegating</strong>, while also identifying <strong>dependencies</strong> and remove <strong>blocks</strong>.</li>
<li>We use <strong>GitHub/ZenHub</strong> and <strong>daily huddles</strong> to assign, scope, prioritize, manage and review our capacity, requirements, estimates - this includes <strong>Project Kanbans, Issue Cards, Pipelines &amp; Reporting</strong>.</li>
</ul>
<p><strong>Team PSD Pain points we are working to better address</strong>:</p>
<ul>
<li>Need &amp;&amp;tight collaboration without unnecessary overhead,** so <strong>culture of participation</strong> is maintained</li>
<li>Need <strong>rapid iterations that prototype,</strong> check early, and <strong>synthesize input</strong>from key stakeholders as efficiently as possible, so we aren’t pulled in wasteful directions or adding unnecessary <strong>rework</strong></li>
<li>Need a <strong>standard set of principles and processes,</strong> so our <strong>production platforms</strong> are <strong>stable at scale.</strong></li>
</ul>
<p><strong>Continuous Collaborative Iteration Cycles (e.g., “DevOps”)</strong></p>
<ul>
<li>SOP &gt; Workgroup Meetings &gt; Monthly Priorities &gt; Requirements gathering standardization for features</li>
<li>Tools for team members to understand cross-functional “stakes” efficiently</li>
<li>“Continuous integration” to avoid “merge conflicts” - <em>Quotes are meant to clarify that we have this problem at a communication and conceptual level, not just a code level.</em> <strong>We need the code level next...because...</strong></li>
<li>Ideally...<strong>testing,</strong> <strong>deploying</strong> and <strong>documenting</strong> would be increasingly automated.</li>
<li>REALLY need continuous documentation and training.</li>
</ul>
